## ModelViewer 架构重写：Blazor/Three.js DOM 彻底隔离 (21:49)

### 问题
ShouldRender 方案仍未能解决"一闪而过变空白"。

### 根因
根本问题是 Blazor 的 DOM diff 引擎与 Three.js 手动管理的 canvas 元素之间的冲突。
即使 ShouldRender 返回 false 跳过渲染周期，但父组件 StateHasChanged 仍可能触发
子组件参数刷新和后续的 DOM 操作，间接影响容器内的 canvas。

### 方案：三层分离架构
将原先"Blazor 管理的 container div 内混杂 canvas + placeholder"的结构，
改为两个完全独立、互不干扰的层：

```
┌─────────────────────────────────┐
│  父级 div (position:relative)    │
│  ┌─────────────────────────────┐│
│  │ Layer 1: js-canvas-host     ││ ← Three.js 独占，Blazor 永不动
│  │ (position:absolute;inset:0) ││   始终存在于模板中（非条件渲染）
│  │ canvas 子元素由 JS append    ││
│  └─────────────────────────────┘│
│  ┌─────────────────────────────┐│
│  │ Layer 2: viewer-placeholder ││ ← Blazor 条件渲染
│  │ (position:absolute;inset:0) ││   ModelUrl 为空时显示
│  │ z-index:10                  ││   ModelUrl 变化时移除
│  └─────────────────────────────┘│
└─────────────────────────────────┘
```

### 关键改动
1. JS canvas 容器 `<div id="@_jsContainerId">` **始终在模板中**，不使用 `@if` 条件
   - Blazor 永远不对这个 div 做 DOM diff（它的子元素只有 canvas，但 canvas 是 JS 添加的非 Blazor 元素）
   - 容器本身不变（id/element 始终一致），Blazor 不会重建它
2. Placeholder 是独立的条件渲染 overlay，z-index:10 覆盖在 canvas 上方
3. 移除 `ShouldRender()` 覆盖（不再需要）
4. 移除旧的 `_containerRef`（不需要 Blazor 引用 JS 容器，直接用 id 选择）
5. CSS 类 `.js-canvas-host` 替代 `.model-viewer`，Only 背景 + overflow 装饰

## GLB vertex interleaving byteStride 缺失修复 (21:57)

### 问题
模型加载日志全绿（[three-module] GLTF loaded successfully）但画面空白。

### 根因
`ProceduralGeometryBuilder.BuildGltfJson` 中 vertex bufferView 缺 `byteStride`:
- Vertex 数据是交错的: `[pos.xyz][normal.xyz][uv.uv]` (每顶点 8×4=32 bytes)
- 但 bufferView 没声明 `byteStride = 32`
- Three.js GLTFLoader 默认紧密排列读取 → 读到的 position 实际为 normal 数据 → 顶点错位
- 所有 object 的顶点落到随机位置，模型不可见

### 修复
`ProceduralGeometryBuilder.cs` 第 420 行 vertex bufferView 加 `byteStride = 32`
